**일시**: 2026-01-17

**주제**: 문서와 코드를 구분하는 Git 전략 및 머지 프로세스 정립

---

## 1. 변경 제안 요약

### 변경 동기 (Why)
문서 변경과 코드 변경의 성격이 다름에도 불구하고 혼재되어 관리되는 문제를 해결하고, 충실한 검토 과정을 거치기 위해 "즉시 머지 금지" 및 "문서/코드 분리" 원칙을 도입함.

### 변경 내용 (What)
1. **Git 전략 (Doc vs Code)**: 
   - 브랜치 접두사를 명확히 구분 (`docs/`, `feat/`, `fix/`, `research/`).
   - 문서만 수정하는 경우 `docs/` 브랜치를 사용하여 코드 빌드 파이프라인과의 간섭 최소화.
2. **즉시 머지 금지**: 
   - 모든 변경은 PR(Pull Request)을 거쳐야 하며, 최소 1명 이상의 페르소나 또는 사용자의 명시적 승인 후 머지.
3. **전문 티켓팅**: 
   - "문서 발전 전략"과 같은 장기적인 문서화 과업을 별도의 백로그 아이템으로 관리.

### 예상 영향 (Impact)
- **긍정**: 변경 이력의 명확성 증대, 코드 품질 및 문서 정확도 향상.
- **부정**: 단순 오타 수정 등 가벼운 변경에도 PR 절차가 요구되어 속도가 다소 느려질 수 있음.

## 2. 페르소나 검토 (3라운드)

### 라운드 1: 독립 의견

#### Senior Software Engineer (시니어 개발자)
**입장**: 찬성  
**의견**:
- `feat/`와 `docs/` 브랜치를 나누는 것에 전적으로 찬성함. 코드 리뷰 시 문서 수정이 섞여 있으면 로직에 집중하기 어려움.
- "즉시 머지 금지"는 CI/CD 안정성을 위해 필수적임.

#### Research Lead (연구 리드)
**입장**: 찬성  
**의견**:
- `research/` 브랜치를 통해 실험적인 코드와 그에 따른 연구 문서(`ai-rules/tasks/research.md` 등)를 묶어서 관리하고 싶음.
- 머지 전 검토 단계가 연구 결과의 신뢰성을 높여줄 것임.

#### System Architect (시스템 아키텍트)
**입장**: 강력 찬성  
**의견**:
- 아키텍처 문서는 코드의 설계도이므로, 코드가 바뀌기 전에 문서가 먼저 혹은 동시에 PR로 올라와야 함.
- 머지 전 "Conflict" 체크와 "Spec 준수" 여부를 검증하는 프로세스가 강화될 것임.

#### Project Manager (프로젝트 매니저)
**입장**: 강력 찬성  
**의견**:
- "문서 발전 전략"을 별도 티켓으로 끊는 아이디어가 매우 좋음. 문서는 부차적인 작업이 아니라 프로젝트의 핵심 자산으로 다뤄져야 함.

---

### 라운드 2: 반박 및 보완

#### Senior Software Engineer
- 문서 브랜치(`docs/`)에서도 최소한의 린트 체크는 돌아가야 함. 오타나 깨진 링크가 `main`에 들어가는 걸 막아야 함.

#### Project Manager
- 머지 승인 주체를 명확히 하자. 문서 변경은 PM이나 아키텍트가, 코드 변경은 개발 리드가 주로 검토하는 식으로 역할을 나누면 효율적임.

#### System Architect
- `docs/` 브랜치는 배포 파이프라인을 트리거하지 않도록 설정하여 CI 비용을 절감할 수 있음.

## 3. PM 최종 보고서

### 합의 사항
- **브랜치 구분**: `feat/` (기능), `fix/` (버그), `docs/` (문서), `research/` (연구).
- **머지 정책**: 즉시 머지 엄금. PR 오픈 후 명시적 승인(Approved) 필수.
- **문서화 전담**: 문서 개선은 개발 작업의 일부가 아닌 독립된 과업(Ticket)으로 정의함.
