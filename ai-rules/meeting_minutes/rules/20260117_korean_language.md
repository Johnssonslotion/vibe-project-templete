# AI-Rules 회의록: 한글 기본 원칙 도입

**일시**: 2026-01-17  
**참석자**: [페르소나 기반 검토] Senior Software Engineer, Research Lead, System Architect, Project Manager  
**주제**: 모든 문서를 한글로 작성하도록 AI Rules 추가

---

## 1. 변경 제안 요약

### 변경 동기 (Why)
프로젝트 문서의 일관성과 협업 효율성을 높이기 위해 모든 산출물을 한글로 작성하도록 통일

### 변경 내용 (What)
`core_principles.md`에 새로운 원칙 추가:  
**"5. Korean Language (한글 기본)"**

- 모든 프로젝트 문서, 회의록, 코드 주석, README는 한글로 작성
- 예외: 코드 자체(변수명, 함수명), 외부 오픈소스 기여, 국제 협업 시 영어 허용

### 예상 영향 (Impact)
- **긍정**: 팀 내 의사소통 명확화, 문서 작성 속도 향상
- **부정**: 영어권 협업자 접근성 감소 (현재 프로젝트는 국내 중심이므로 영향 적음)

## 2. 페르소나 검토 (3라운드)

### 라운드 1: 독립 의견

#### Senior Software Engineer (시니어 개발자)
**입장**: 조건부 찬성  
**의견**:
- 한글 문서화는 국내 팀 협업에 효율적이나, 코드베이스가 글로벌 오픈소스로 확장될 가능성 고려 필요
- 예외 조항("외부 오픈소스 기여, 국제 협업 시 영어 허용")이 명확하므로 실용적
- 코드 주석까지 한글 강제는 IDE 호환성 문제 발생 가능 (일부 도구는 비ASCII 주석에서 인코딩 오류)
- **조건**: 코드 주석은 "권장"으로 완화, README는 이중 언어(한/영) 제공 옵션 추가

#### Research Lead (연구 리드)
**입장**: 찬성  
**의견**:
- 연구 문서(실험 계획, 결과 분석)를 한글로 작성하면 사고 속도와 기록 속도가 일치해 생산성 향상
- 논문 작성 시에는 영어가 필수이지만, 내부 문서는 한글이 더 효율적
- 페르소나 토론이나 회의록도 한글로 작성되고 있으므로 일관성 확보에 긍정적
- **의견**: 규칙에 "단, 학술 출판물은 영어 작성" 예외 명시 필요

#### System Architect (시스템 아키텍트)
**입장**: 조건부 찬성  
**의견**:
- CI/CD 로그, 에러 메시지, 시스템 문서는 영어가 표준이므로 한글 강제 시 통합 문제 발생
- Dockerfile, YAML 설정 파일 주석은 국제 표준을 따르는 것이 유지보수에 유리
- 프로젝트 governance 문서는 한글 우선이 맞지만, 인프라 코드는 예외 인정 필요
- **조건**: "인프라 코드 및 설정 파일은 영어 주석 허용" 조항 추가

#### Project Manager (프로젝트 매니저)
**입장**: 찬성  
**의견**:
- 현재 팀 구성이 모두 한국어 사용자이므로 한글 통일이 합리적
- 문서 작성 장벽을 낮춰 팀원들의 기여를 높일 수 있음
- 향후 영어권 협업자 참여 시에는 해당 문서만 선택적으로 번역하면 됨 (전체 규칙 변경 불필요)
- **의견**: 버전 관리 측면에서 CHANGELOG는 한/영 병기 고려

### 라운드 2: 반박 및 보완

#### Senior Software Engineer
**타 페르소나 의견에 대한 반응**:
- Research Lead의 "학술 출판물 예외" 동의. 실제로 논문 작성 시 영어 필수이므로 명시 필요
- System Architect의 인프라 코드 예외 동의. Docker, K8s YAML은 영어 주석이 국제 표준
- **보완 제안**: "코드 주석은 한글 권장, 단 오픈소스 기여 또는 인프라 코드는 영어 허용"으로 유연성 확보

#### Research Lead
**타 페르소나 의견에 대한 반응**:
- Senior Engineer의 코드 주석 완화 의견에 일부 동의하나, 일반 애플리케이션 코드는 한글 주석이 가독성 향상
- System Architect의 인프라 예외는 타당함
- **보완 제안**: "애플리케이션 코드는 한글 주석 우선, 인프라/설정 파일은 영어 허용"으로 구분

#### System Architect
**타 페르소나 의견에 대한 반응**:
- PM의 CHANGELOG 한/영 병기는 좋은 아이디어이나, 유지보수 부담 증가 우려
- 대신 한글 CHANGELOG + 영어 Release Notes 자동 생성 도구 활용 제안
- **보완 제안**: "CHANGELOG는 한글 작성, CI에서 주요 릴리즈 시 영어 번역 자동 생성"

#### Project Manager
**타 페르소나 의견에 대한 반응**:
- 모든 페르소나가 "예외 조항"의 필요성을 인정하므로, 규칙을 너무 경직되게 만들지 않는 방향 지지
- System Architect의 자동 번역 제안은 실용적이나, 현 단계에서는 과도한 자동화
- **보완 제안**: "한글 기본 원칙 + 명확한 예외 목록"으로 단순화하고, 자동화는 향후 과제로

## 3. PM 최종 보고서

### 합의 사항
- **한글을 기본 언어로 채택**: 모든 프로젝트 문서, 회의록, README, 일반 코드 주석은 한글로 작성
- **CHANGELOG 이중 언어 전략**: 
  - **영어를 메인**으로 배치 (LLM 토큰 효율성, 국제 표준 준수)
  - **한글 병기**로 사용자 편의 제공
  - 형식: 각 항목을 영어로 먼저 작성 후 한글 번역 추가
- **명확한 예외 조항 명시**:
  1. 오픈소스 기여 및 외부 협업: 영어 사용
  2. 인프라 코드 및 설정 파일 (Dockerfile, YAML, Terraform 등): 영어 주석 허용
  3. 학술 출판물: 영어 작성
  4. 국제 협업 시: 영어 사용

### 보류/반대 사항
없음 (모든 페르소나가 조건부 또는 전면 찬성)

### 리스크 및 완화책
| 리스크 | 완화 방안 |
|--------|-----------|
| 향후 글로벌 협업 시 언어 장벽 | 예외 조항 활용, 필요 시 선택적 영어 번역 |
| 코드 주석 인코딩 문제 | IDE UTF-8 설정 표준화, pre-commit 인코딩 검증 |
| CHANGELOG 이중 언어 유지보수 부담 | 영어 메인 작성 후 한글 번역 추가, CI 자동화는 향후 검토 |

### 최종 권고
**권고안**: 조건부 승인  
**근거**: 모든 페르소나가 한글 우선 원칙에 찬성하며, 실용적인 예외 조항을 통해 유연성 확보 가능. 현재 팀 구성과 프로젝트 성격상 한글 문서화가 생산성 향상에 기여할 것으로 판단.
