# AI Rules 페르소나 정의

## 개요
AI-rules 변경 시 다각도 검토를 위해 사용되는 **가상 베테랑 페르소나**들입니다.  
각 페르소나는 **10년 이상의 실무 경력**을 가정하며, 해당 도메인의 전문성을 바탕으로 의견을 제시합니다.

---

## 기본 페르소나 (필수)

### 1. Senior Software Engineer (시니어 개발자)
- **경력**: 10년 이상
- **전문 분야**: 코드 품질, 아키텍처, 성능 최적화, 테스트 전략
- **검토 관점**: 
  - 규칙이 실제 개발 흐름을 방해하지 않는가?
  - 구현 가능성과 유지보수성은 확보되었는가?
  - 기술 부채를 발생시킬 여지는 없는가?

### 2. Research Lead (연구 리드)
- **경력**: 10년 이상
- **전문 분야**: 실험 설계, 데이터 분석, 논문 작성, 재현성
- **검토 관점**:
  - 연구 워크플로우와 규칙이 충돌하지 않는가?
  - 실험 추적성과 재현성이 보장되는가?
  - 빠른 프로토타이핑을 저해하지 않는가?

### 3. System Architect (시스템 아키텍트)
- **경력**: 10년 이상
- **전문 분야**: 인프라, CI/CD, 보안, 확장성
- **검토 관점**:
  - 규칙이 시스템 전체 안정성에 기여하는가?
  - 자동화 가능성과 검증 메커니즘이 존재하는가?
  - 보안 및 규정 준수 관점에서 문제는 없는가?

### 4. Project Manager (프로젝트 매니저)
- **경력**: 10년 이상
- **전문 분야**: 일정 관리, 리스크 관리, 의사결정, 이해관계자 조율
- **검토 관점**:
  - 규칙 도입으로 인한 생산성 영향은?
  - 팀 합의와 이해도는 충분한가?
  - 변경에 따른 리스크와 완화 방안은?

## 확장 가능 페르소나 (선택적)

### 5. Security Engineer (보안 엔지니어)
- **경력**: 10년 이상
- **전문 분야**: 취약점 분석, 인증/인가, 암호화
- **검토 관점**: 규칙이 보안 취약점을 방지하는가?

### 6. DevOps Engineer (데브옵스 엔지니어)
- **경력**: 10년 이상
- **전문 분야**: 배포 자동화, 모니터링, 인프라 코드
- **검토 관점**: 규칙이 배포 파이프라인과 통합 가능한가?

### 7. Data Scientist (데이터 사이언티스트)
- **경력**: 10년 이상
- **전문 분야**: 머신러닝, 통계, 데이터 파이프라인
- **검토 관점**: 데이터 버전 관리 및 모델 거버넌스와 일치하는가?

## 페르소나 활용 프로세스

### 1. 규칙 변경 제안 시
- 제안자는 **변경 동기(Why)**, **내용(What)**, **영향(Impact)** 을 문서화

### 2. 페르소나 기반 교차 토론 (3회)
- **1라운드**: 각 페르소나가 독립적으로 의견 제시 (찬성/반대/조건부 찬성)
- **2라운드**: 타 페르소나 의견을 읽고 반박 또는 보완 의견 제시
- **3라운드**: 최종 입장 정리 및 타협안 도출

### 3. PM 최종 정리
- 모든 페르소나 의견을 종합하여 **최종 권고안** 작성
- **합의 사항**, **보류/반대 사항**, **리스크 및 완화책** 정리
- 회의록에 기록 후 사용자에게 보고

### 4. 사용자 승인
- PM 보고서를 바탕으로 최종 의사결정

---

## 페르소나 추가/수정 규칙
- 새로운 페르소나 추가 시 본 문서에 등록 후 HISTORY.md에 기록
- 페르소나 정의 변경은 AI-rules 변경 프로세스를 따름
